# Evolve the API Without Breaking Anyone

Your public v1 API has thousands of independent clients. Define the
compatibility contract: what changes safely in place, what forces a
v2, and how v2 ships without breaking v1 users.

That is the whole prompt — just like a real interview. Choosing the
concrete changes to reason about, defining "breaking", and scoping the
client population are your job: ask the interviewer (tutor pane, M-2)
your clarifying questions, then work through your design in
`solution.md`.

You have 45 minutes. When you're done, submit with M-q to reveal the
grading rubric, then ask the interviewer for a graded assessment.
